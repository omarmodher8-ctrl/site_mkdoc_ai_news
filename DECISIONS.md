# DECISIONS

## 2025-09-18 - 依存パッケージのバージョン固定を行わない
- `requirements.txt` はバージョン未指定のまま運用する。
- Cloudflare Pages のビルドで異常が起これば検知できるため、その時点で対処する。
- 必要になった場合のみ、直前に動作したバージョンをメモしつつ段階的に固定する方針。

## 2025-09-18 - `.memo` ファイルの二重読み込みを維持
- プラグインとマクロの両方で `.memo` を読む冗長構成はそのままにする。
- 冗長でも安全側に倒しておきたいという意図で、パフォーマンス影響は小さい。

## 2025-09-18 - カテゴリ表示名のハードコードをフォールバックとして残す
- `macros/home.py` の `CATEGORY_BASE_LABELS` はナビゲーションのタイトル取得が失敗した際の保険とする。
- `.pages` でタイトルを定義できるが、ナビゲーション情報が得られないケースに備えて残置。
- コメントで意図を明記済み。

## 2025-09-18 - `.memo` はAIワークフローで自動生成する予定
- 記事本文からの `.memo` 生成は Gemini による既存フローを転用する。
- すでに別サイトで 5000 件以上の `.memo` を自動生成した実績があるため、人手管理の懸念は小さい。
- そのため `.memo` を前提とした description 取得の仕組みはこのまま維持する。

## 2025-09-18 - 記事概要は `.memo` で外部化し再利用性を確保
- `.md` に description を埋め込むと箇所によって個別パーサーが必要になる。
- `.memo` を使えば OGP／JSON-LD／トップページなど複数用途で流用できる。
- そのため概要テキストは `.memo` に分離する運用を続ける。

## 2025-09-18 - パフォーマンス最適化はビルド時間が問題化したら検討
- 想定記事数は長期でも千件前後であり、現行マクロや `.memo` 読み込みの負荷は軽微。
- MkDocs 自体が大量ページ（数十万件）を想定した静的ジェネレータではないため、過剰最適化は行わない。
- ビルド時間に支障が出た時点で JSON キャッシュ等の対策を再検討する。

## 2025-09-18 - 年次アーカイブへの移行は需要発生時に手動または簡易スクリプトで対応
- カテゴリは当面 2 本の想定であり、2025→2026 などの区切りで手動移動でも十分。
- 実際にフォルダが増えてから年別整理の要件を確定し、自動化が必要ならスクリプトを追加する。
- 2026 年中頃を目安に状況を見て判断する。
