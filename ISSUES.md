# プロジェクトの問題点分析レポート

このドキュメントは、当プロジェクトのコードベースを分析し、特定された問題点とその改善提案をまとめたものです。

---

## 1. 環境構築に関する問題

### 問題点: セットアップ手順のドキュメント不足

- **現状**: `README.md`には `pip install -r requirements.txt` の実行については示唆されていますが、カスタムプラグイン `description_from_memo` を有効にするために必須である `pip install -e .` コマンドについての記述がありません。
- **問題**: 新しい環境でプロジェクトをセットアップする際、この手順を省略すると `mkdocs serve` 実行時に `PluginError` が発生し、ビルドに失敗します。開発者本人であっても、時間が経ってから環境を再構築する際に手順を忘れる可能性があります。
- **推奨される解決策**: `README.md`に、`pip install -r requirements.txt` の次に `pip install -e .` を実行する必要があることを明記します。

---

## 2. 依存関係に関する問題

### 問題点: 依存関係のバージョンが固定されていない

- **現状**: `requirements.txt` に記載されているライブラリ (mkdocs, mkdocs-materialなど) のバージョンが指定されていません。
- **問題**: `pip install` を実行するタイミングによって各ライブラリの異なるバージョンがインストールされるため、環境の再現性が保証されません。ライブラリのアップデートで破壊的変更があった場合、予期せずサイトがビルドできなくなるリスクがあります。
- **推奨される解決策**: `pip freeze > requirements.txt` を実行し、現在正常に動作している環境のライブラリバージョンを `requirements.txt` に固定します。これにより、誰がいつセットアップしても同じ環境を構築できます。

---

## 3. 設定に関する問題

### 問題点: OGP画像へのパスが正しくない

- **現状**: `mkdocs.yml` の `extra.default_image` で `assets/images/default_ogp.png` が指定されていますが、このパスにファイルが存在しません。
- **問題**: サイトのリンクをSNSなどで共有した際に、意図したOGP画像が表示されません。
- **推奨される解決策**:
    - a) `assets/images/default_ogp.png` というパスに画像ファイルを配置する。
    - b) `mkdocs.yml` のパスを、実際に使用したい画像の正しいパスに修正する。
    - c) OGP画像を意図的に使用しないのであれば、`extra.default_image` の行を削除する。

---

## 4. コード品質に関する問題

### 問題点: `.memo` ファイルの読み込みロジックの重複

- **現状**: `plugins/description_from_memo.py` (プラグイン) と `macros/home.py` (マクロ) の両方で、`.memo` ファイルを読み込む処理が実装されています。
- **問題**: 同じ処理が2箇所に存在するため、コードが冗長になっています。将来的に `.memo` ファイルの処理方法を変更する場合、2箇所を修正する必要があり、修正漏れやバグの原因となります。また、1つのページ表示のために同じファイルが2回読み込まれる可能性があり、非効率です。
- **推奨される解決策**: ロジックを一元化します。プラグインを「`.memo` ファイルからdescriptionを読み込み、`page.meta` に設定する唯一のコンポーネント」と位置づけます。マクロ側では `.memo` ファイルを直接読まず、プラグインが設定した `page.meta.description` を常に参照するように修正します。

## 5. 保守性に関する問題

### 問題点1: カテゴリ表示名がハードコードされている

- **現状**: `macros/home.py` 内の `CATEGORY_BASE_LABELS` という辞書で、URLのslug（例: `ai-generated-news`）と日本語の表示名（例: `生成AIニュース`）がマッピングされています。
- **問題**: 新しいカテゴリのディレクトリを追加した場合、この辞書を手動で更新しない限り、意図したカテゴリ名で表示されません。メンテナンス忘れが発生しやすい箇所です。
- **推奨される解決策**: `awesome-pages`プラグインが生成するナビゲーションのタイトル (`page.ancestors[0].title`) を活用することで、カテゴリ名の一部を自動で取得できる可能性があります。現在の実装でもフォールバックとして既に使われているため、このロジックを主軸に据えることで、ハードコードされた辞書の更新頻度を減らせます。

### 問題点2: 日付処理でタイムゾーンが考慮されていない

- **現状**: マクロ内で `datetime.fromisoformat()` や `strftime()` を使用する際に、タイムゾーン情報が扱われていません。
- **問題**: ニュース記事において日付は重要な情報ですが、タイムゾーンが考慮されていないと、ビルド環境や閲覧環境によって日付の解釈がずれる可能性があります。
- **推奨される解決策**: 日付を扱う際は、`datetime.now(timezone.utc)` のようにタイムゾーンを明示的に指定するか、すべての時刻をUTC（協定世界時）として扱うように統一することが望ましいです。

### 問題点3: 未使用の可能性があるマクロ

- **現状**: `macros/home.py` には `latest_pages` と `home_sections` の2つのマクロが定義されていますが、`docs/index.md` で使用されているのは `home_sections` のみです。
- **問題**: `latest_pages` マクロは現在使用されておらず、デッドコード（不要なコード）である可能性があります。コードベースが複雑になり、メンテナンスコストを増大させます。
- **推奨される解決策**: プロジェクト全体で `latest_pages` が本当に使われていないことを確認した上で、`macros/home.py` からこのマクロの定義を削除します。
