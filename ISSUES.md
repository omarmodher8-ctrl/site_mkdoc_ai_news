# プロジェクトの課題 (Project Issues)

このドキュメントは、`site_mkdoc_ai_news` プロジェクトの現状分析に基づき、考えられる課題や改善点をまとめたものです。

---

## 1. `.memo` ファイルの管理に関する課題

現状、各記事ページ（`.md`）の `description`（概要）は、同名の `.memo` ファイルから取得する仕組みになっています。これには以下の課題が考えられます。

- **管理コストの高さとヒューマンエラーのリスク**:
  - 記事（`.md`）とは別に概要（`.memo`）をファイルとして管理する必要があり、手間が二重になっています。
  - Grokによる記事生成プロセスで `.memo` ファイルが自動生成されない場合、手動での作成・更新が必要となり、以下のようなヒューマンエラーが発生しやすくなります。
    - `.memo` ファイルの作成漏れ
    - `.md` ファイル更新時の `.memo` ファイルの更新漏れ
    - ファイル名のタイポ
- **コンテンツの散在**:
  - 本文と概要が別ファイルに分離しているため、コンテンツの全体像が把握しにくく、管理が煩雑になります。

### 提案

記事の概要は、各 `.md` ファイルの先頭にYAMLフロントマター形式で直接記述することを推奨します。

**例 (`YYYY-MM-DD.md`):**
```markdown
---
date: 2025-09-17
description: ここに記事の概要を記述します。
---

# 記事のタイトル

本文...
```

これにより、コンテンツとメタデータが一元管理され、`.memo` ファイルは不要になります。

## 2. メタデータ抽出処理の冗長性

トップページの生成マクロ (`macros/home.py`) とカスタムプラグイン (`plugins/description_from_memo.py`) の両方で、メタデータを取得するためのロジックが実装されており、処理に冗長性が見られます。

- **`description` 取得ロジックの重複**:
  - `plugins/description_from_memo.py` は `.memo` ファイルを読み込み、ページの `meta.description` を設定します。
  - 一方で `macros/home.py` 内の `extract_page_info` 関数も、`meta.description` が無い場合に備えて、再度 `.memo` ファイルを読み込むフォールバック処理を持っています。
  - プラグインが常に先に実行されるため、マクロ側の処理は現状ほぼ実行されませんが、コードの重複と複雑化を招いています。
- **`title` 取得ロジックの複雑性**:
  - `title` を取得するために、`page.meta` -> YAMLフロントマター -> MarkdownのH1見出し -> ファイル名、という多段階のフォールバック処理が存在します。
  - この複雑性は、意図しないタイトルが表示される原因となったり、問題発生時のデバッグを困難にしたりする可能性があります。

### 提案

- メタデータの取得元をYAMLフロントマターに統一し、`description_from_memo` プラグインと `.memo` ファイルを廃止します。
- これに合わせて `macros/home.py` のロジックを簡素化し、フロントマターからのみメタデータを取得するように修正します。

## 3. 将来的なパフォーマンスへの懸念

トップページに最新記事一覧などを表示するマクロ (`home_sections`, `latest_pages`) は、サイトの全ページを一度に読み込んでからソート・フィルタリングする設計になっています。

- **ビルド時間の増加**:
  - 現在のページ数では問題にならないかもしれませんが、今後、記事数が数千、数万と増加した場合、サイトのビルド時間（`mkdocs serve` の起動や `mkdocs build` の実行時間）が著しく長くなる可能性があります。
  - 開発中のプレビュー更新が遅くなったり、デプロイに時間がかかったりする原因となります。

### 提案

- **（中長期的対応）インデックスファイルの活用**:
  - ビルド時に全ページのメタデータ（日付、タイトル、概要、URL）をまとめた単一のJSONファイル（例: `metadata.json`）を生成するようなプラグインを導入または作成します。
  - マクロは全ページを読み込む代わりに、このJSONファイルのみを読み込んでトップページを生成するようにします。これにより、ページ数が増加してもビルドパフォーマンスへの影響を最小限に抑えることができます。
